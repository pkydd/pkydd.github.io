    Workshop 0: Getting started

    Overview:

    This workshop is designed to help you set up a machine with Linux and ROS2 in preparation for the following workshops.

    You will need:  To complete the initial workshops, you will really only need a PC and a little bit of space (~40GB should be enough).

    The following sections will help guide you through the set up process for both Linux and ROS2. 

    Time to complete: < 1 Hour

    Setting up a Linux install

    Firstly, we will need to set up our computers so they are robo-ready. Linux is the best (supported) option here, but if you are feeling particularly daring, you might like to try Windows or OSX (warranty void etc). Linux is an open source operating system that functions more or less like Windows/OSX - we’ll talk about that more as we progress through the workshops.

    ELI5: Operating Systems, Linux and Open Source An operating system (OS) is a type of software that runs on your computer that manages your hardware and software, acting as a resource manager while your computer is running. It provides (ideally) an interface that makes the user <-> computer interaction easier, as well as allowing the user to run applications on the machine in a convenient way. When you run Google chrome on your computer, it is actually an application that is running on your operating system, which is running on your computer hardware.

    Windows, OSX and Linux are all examples of operating systems - they just work slightly differently and offer different capabilities and experiences. OSX is elegant and easy to use, Linux is open source and extremely flexible and Windows does its best. Linux is particularly useful for programming and other technical work, since it (through the community) provides a robust set of tools to work with that make computing tasks a little more straightforward and dare I say, enjoyable?

    “Open source” is a term describes software (and sometimes hardware) that is available for anyone to access and read. For example, if you wanted to, you could look at the code that is used to create the video player VLC here: https://github.com/videolan/vlc - the source is literally open to be inspected. However, you cannot (easily) look at the equivalent code for the OSX or Windows built in media players - we call this “closed source”. Typically open source projects are open for community contribution, but sometimes companies make their code open source too.

This can be particularly useful for spinning up quick instances of an OS for testing, or simply for convenience.     A note on virtual OS’s: when running a virtual OS, your “stack” will look something like this:

    The virtual machine is running alongside your other apps on your Host OS (Windows in this case), with Linux running on top of it, and open solitaire on top of that.

    This is obviously super convenient, but we typically compromise on performance when running in this configuration. In the above diagram, you can see that we have added another “level” of interfaces (Virtual Machine and Linux) that applications need to call through to “reach” the hardware layer. Previously, when Solitaire running on our host OS wanted to access your machine hardware, it would do so through the Windows interface layer. OpenSolitaire running on linux however, has several additional layers to “call” through. This additional layer (among other things) can certainly slow virtual machines down quite a bit.

    Option 2: Baremetal/Dualboot

    We can also install Linux directly onto our machine, no virtualisation required! This is a little trickier, particularly if you are intending to dualboot with Windows. At this point the IT department probably wants me to advise against doing this on your work machines.

    If you are installing on a blank machine and not intending to dualboot, its easy:

    1. Download ISO
    2. Download Rufus - this is a tool that allows the easy creation of bootable USB drives
    3. Create bootable USB stick. plug it in and restart. Note: you may need to change your BIOS boot order to access the USB stick.
    4. Follow the Ubuntu/Xubuntu installation prompts.
    5. Enjoy

    Dualbooting is a little trickier, but basically:

    1. Create bootlable USB as above
    2. In your Windows settings, disable fastboot:

    1. In Windows, create a partition of appropriate size:

    1. Boot into your BIOS and disable Secureboot.
    2. Boot from your Bootable USB stick
    3. Install as above, but take care that you select your efi partition for your boot loader. You should see somethng like this:

    1 is your bootloader partition, 2 is your Windows partition, 3 is soon to be your Linux partition.( + swap space )

    Make sure you have your boot partition set, a swap space set up and the remaining space set to be the root of your filesystem. There is more detail here, but I don’t want to tangent too much. Don’t wipe your windows partition and anything else is more or less recoverable.

    ELI5: Packages? what?

    Ok, so another core difference between Windows and Linux is that the various Linux (distributions - “distros”, such as ubuntu, Debian, Arch etc) maintain a set of packages that are available as part of their “Package Manager” system. Anyone who has used python’s pip before will be familiar with this, but basically a package is a set of libraries or programs that do a certain thing.

    In Windows, if you want to install a program, you usually google the name of the program and try and find the appropriate installer (like python, for example). You get the .exe, run it, install the program into who knows where in your directory structure, and then when you want to run it, you go and find the exe and double click it.

    Linux distros (like Ubuntu) tend to be a bit more organised. As I mentioned, you have these package lists that are maintained officially, and when you want to download something, like python, you simple ask the package manager to get the package and install it (like pip..ish):

    sudo apt install gcc

    The caveat here is that if you have older package lists, you might download something that is out of date. Also, not everything is stored in the “official packages”. Finally, while the system does attempt to make sure things are compatible, you can definitely force the system to do the wrong thing. (Uncommon through normal use, typically the package manager will warn you that something is wrong).

    Once the package is installed, the relevant files will be placed in a sane location on your system. That’s right, unlike Windows where you have things install where ever they please, Linux tends to try and keep all your binaries, libraries etc in the same place (typically the root directory, /bin, /sbin, /opt etc depending on what type of package you have installed).

    Installing ROS2

    Ok, so now we will install ROS2 onto our machines. If you want to skip all my waffling, you can follow the very excellent instructions here:

    ROS2 Installation

    Otherwise, in your terminal, run the following commands:

    locale  # check for UTF-8

sudo apt update && sudo apt install locales
sudo locale-gen en_US en_US.UTF-8
sudo update-locale LC_ALL=en_US.UTF-8 LANG=en_US.UTF-8
export LANG=en_US.UTF-8

locale  # verify settings

    This updates our locale, which is basically just the environment variables that tell the system which character set, language etc to use.

    sudo apt update 
sudo apt install curl gnupg lsb-release
sudo curl -sSL https://raw.githubusercontent.com/ros/rosdistro/master/ros.key  -o /usr/share/keyrings/ros-archive-keyring.gpg

    We have already seen the update (and executed it, so we don’t really need to run it again here, but it doesn’t hurt). The next line installs a few useful packages, and the last line uses a program called curl to update our distro package list keys.

    Next we add the package repository to our lists and off we go:

    echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/ros-archive-keyring.gpg] http://packages.ros.org/ros2/ubuntu $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/ros2.list > /dev/null

    Then:

    sudo apt update
sudo apt install ros-galactic-desktop

    And wait for that to finish. Once it is done, you should be able to test your install!

    Firstly, lets make our lives a little easier:

    There is a file called setup.bash, that allows us to set a bunch of environment variables that we will want to set to be able to use ROS2. We need to set that in each terminal that we run, however, we can add it to the environment set up so that it is automatically set each time we open a new terminal.

    Copy the line: source /opt/ros/galactic/setup.bash

    And add it to the end of your .bashrc file, in your home directory:

    /home/$username/.bashrc

    and now, open two new terminals. (or run the source command on the above file from existing terminals).

    In one, run:

    ros2 run demo_nodes_cpp talker

    and the other run:

    ros2 run demo_nodes_py listener
